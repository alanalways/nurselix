- [ ] **Step 3: Result endpoint computes tier**

```typescript
function classifyTier(theta: number, englishScore: number): "A" | "B" | "C" {
  if (theta >= -1.5 && englishScore >= 0.65) return "A";
  if (theta >= -2.5 || englishScore >= 0.5) return "B";
  return "C";
}
```

`englishScore` derived from: % of questions answered correctly where `englishGap=true` (computed during the session — questions with high vocab difficulty).

**Risk: warranty refunds eat cash**
Mitigation: don't market PRO + warranty heavily until you have run 3 successful test claims internally (use admin-created synthetic users).

**Success criteria for M3 → exit M1-M3 cycle:**
- 10+ paying customers (any tier)
- Monthly recurring revenue (MRR) ≥ NT$10,000
- At least 1 successful warranty claim processed cleanly
- Churn < 20% over first 2 months
- Conversion rate (free → paid) ≥ 5%

If above met → continue to M4 (scaling); if not → re-brainstorm based on what specifically failed.

## Rollback strategy for M3

**Before each task:** git tag `pre-m3-task-NN-<date>`. Run `pg_dump` equivalent JSON snapshot.

**Critical safe-guards:**
- NewebPay live mode toggle (env var `NEWEBPAY_LIVE=false` defaults). Keep sandbox usable for testing.
- WarrantyClaim approval requires admin email confirmation (no API-driven auto-approval).
- Refund button in admin UI requires typing "REFUND" to confirm.

**Worst-case rollback:**
1. Disable signups (`SIGNUPS_DISABLED=true`)
2. Send all paid users a courtesy email
3. Pause NewebPay (live=false)
4. Restore last known good DB snapshot from `.backups/`
5. Investigate root cause before re-opening
